# 越境コーチ (Embedded Exam Coach) — 要件定義

> 本書は本プロトタイプ（HTML/React 試作）の **要件定義** である。
> 年間計画 `uploads/plan.md`（以下「**計画書**」）に記載されたスケジュール・指標・ルールはここに重複させない。
> 計画書の節番号（§4, §5.1 など）を参照する形で扱う。

---

## 1. 本アプリの位置づけ

本アプリは「計画書」を**運用するためのツール**である。計画書は紙の運用でも成立する内容だが、本アプリは以下の3つを機械化することで日々の運用負荷を下げることを目的とする。

1. **今日やることの提示**（日次のタスク確定）
2. **学習結果の記録と集計**（指標・誤答・弱点）
3. **レビュー時の意思決定支援**（週次・月次の判断材料の提示）

「計画を立てる」役割は持たず、計画書を所与のものとして扱う。計画書の編集機能はスコープ外（将来検討）。

## 2. 対象ユーザー

- 計画書の作成者本人、単独利用。
- マルチユーザー / 共有 / 認証は持たない。

## 3. スコープ

### 3.1 In Scope

| 領域 | 内容 |
|---|---|
| 日次運用 | 当日タスクの提示・実行・採点・記録 |
| 科目A-2 | 通し番号制カードの導入/復習スケジュール表示と問題実施（§5.1 のルールに基づく） |
| 科目B-1 | 対象問題（§4.3）に対する実施状況の管理 |
| 科目B-2 | 対象問題（§4.1, §4.2）× 10工数（§5.3）の進捗管理 |
| 指標集計 | 週次指標（§6.1）の自動算出と4週推移の可視化 |
| 弱点管理 | 弱点メモの作成・分類（§7.3）・状態遷移 |
| レビュー | 週次（§6）・月次（§7）の入力支援とテンプレ化 |
| スマホ対応 | 閲覧と記録（タスク完了、セルフ採点、気づきメモ、ミス記録など）をスマホにも提供 |

### 3.2 Out of Scope

- 計画書そのものの編集（§3 の工数試算や §4 の対象問題の変更）
- 問題本文・解説のオーサリング機能をアプリ内に設けること（読み取り専用とし、データは外部データソースを参照する）
- マルチユーザー / 認証 / クラウド同期
- オフライン動作保証（ただしローカル保存採用のため、読み込み後は実質オフラインで動作しうる）

## 4. 用語

計画書で定義された用語（通し番号 c、状況組み立て、設問1〜3 など）は計画書を参照する。本アプリで独自に導入する用語のみここに記す。

| 用語 | 意味 |
|---|---|
| **タスク** | 本日実施する単位。`{ 科目, カード番号 or 問題, 復習回数, 推定所要分 }` を持つ |
| **セッション** | 本日のタスクを一通り消化する一連の作業 |
| **カード進行表** | A-2の通し番号制（§5.1）を 4週間グリッド + 16年×25問マトリクスで可視化したビュー |
| **気づきメモ** | タスク実行中に書く一時的メモ。`弱点メモに昇格` 操作で恒久化される |
| **弱点メモ** | 弱点マップ（§7.3）上の1エントリ。状態 `未着手 / 改善中 / 停滞 / 解消` を持つ |
| **ミス分類** | §6.2 の3カテゴリ（知識不足型・理解不足型・ケアレス型）でのタグ付け |

## 5. 画面一覧と責務

| ID | 画面 | 目的 | 主な計画書参照 |
|---|---|---|---|
| S1 | ダッシュボード | 「今、自分はどこにいるか」を10秒で把握する作戦盤 | §2, §6.1 |
| S2 | 今日のセッション | 今日のタスクキューを順に消化する作業画面 | §5.1〜5.3 |
| S3 | 問題画面 (A-2) | 1枚のカードに対する問題本文・選択肢・採点・解説 | §5.1 |
| S4 | A-2 カード進行 | 通し番号と復習日の予定・実績を俯瞰する | §5.1 |
| S5 | 過去問進捗 | B-1（16年×3問）と B-2（23テーマ×10工数）の消化状況 | §4.1〜4.3, §5.2, §5.3 |
| S6 | 週次レビュー | §6 のフローを「測る→ふりかえる→うごかす」3ステップで誘導 | §6 |
| S7 | 月次レビュー | §7 のフロー（トレンド・到達度・残工数・やる/やらない）を誘導 | §7 |
| S8 | 弱点マップ | 弱点メモを4状態カンバンで管理 | §7.3 |

### 5.1 S1 ダッシュボード

- **必須要素**
  - 試験日までの残日数 / 計画 Day N / N (計画書 §3 の稼働日数)
  - 今日のタスクの達成状況（件数 / 進捗率 / 科目内訳）
  - 連続学習日数 / 今週の学習時間
  - 週次指標の現状（§6.1 の指標、目標値との対比、4週スパークライン）
  - ミスのパターン分布（直近4週）
  - 弱点マップの状態別件数（解消 / 改善中 / 停滞 / 未着手）
  - 直近の学習記録（§8 のテンプレートに沿った形）
- **コーチメッセージ枠**：今日の重点や警告を1〜2文で出す領域（生成ロジックは将来検討、現状はモック）

### 5.2 S2 今日のセッション

- **タスクキュー**：本日のタスクを上から順に表示。クリックで「現在の課題」に昇格。
- **現在の課題カード**
  - タスク内容（科目タグ、原問の表記、復習回数）
  - A-2 タスクの場合：通し番号・原問（§5.1 ⓪逆引き）・復習間隔（§5.1 ②）・次回出現
  - 操作：`問題を開く` (A-2のみ → S3)、`完了`、`スキップ`
- **気づきメモ**：自由記述。`+ 弱点メモに昇格` で S8 に転送
- **セルフ採点**
  - A-2：`正答 / 迷った / 誤答` の3択。選択結果に応じて次回出現の挙動メッセージを表示
  - B-1：設問1〜3 の ○△× ＋ 所要時間
  - **B-2 の4観点採点はここでは行わない**（§6.2 の運用上、週次レビュー S6 で実施）
- **ミス記録**：3カテゴリのカウンタ（§6.2）。誤答時に増減

### 5.3 S3 問題画面 (A-2)

- **入口**：S2 の `問題を開く` または S4 のミニカード/カード詳細から遷移
- **要素**
  - 問題ヘッダ（通し番号 / 原問 / 復習回数 / 経過秒）
  - 問題本文（外部データソースから読み込み）
  - **図を含む問題本文**の表示（設問中に図が含まれる、選択肢それ自体が図というケースもあるものとしてレイアウトを設計する）
  - 選択肢 ア・イ・ウ・エ（テキストまたは画像）
  - 解答ボタン → 提出後に正誤判定と解説表示
  - 関連トピックタグ
  - サイドにこのカードのメタ（導入日 Day, 次回出現 +N日）
  - 誤答時にのみ有効化されるミス分類タグ
  - 気づきメモ
- **データ**：問題本文・選択肢・解説・図（画像）は **外部データソース** から読み込む。詳細は §6 データモデルの `Problem` エンティティを参照。
- **コンテンツ未整備時の振る舞い**：問題本文が未登録のカードでも画面は開き、「本文未取り込み」と示しつつ以下の記録だけは可能にする：
  - 正誤・所要時間・ミス分類・気づきメモ

### 5.4 S4 A-2 カード進行

- **4週間スケジュールグリッド**：1セル = 1日。その日に学ぶカード番号をミニカード（導入 ✦ / 復習回数別の色）で表示。
- **ミニカード操作**：クリック → 問題画面 S3 へ。Shift+クリック → カードのフォーカスのみ。
- **カード詳細**：選択中カードの導入日・出現スケジュール（公比2、§5.1 ②）・完了履歴
- **16年×25問マトリクス**：通し番号と原問の対応を視覚化（§5.1 ⓪）。クリックでフォーカス変更
- **本画面では復習スケジュールの計算式・ペース等を表示するが、計算式そのものは§5.1の式を参照表示する形とする。**

### 5.5 S5 過去問進捗

- **B-2 ビュー**
  - SA出題14テーマ（§4.1）と ES出題9テーマ（§4.2）を分けて表示
  - 各テーマを 10 工数のセル（§5.3：状況組み立て1〜7 + 設問1〜3）で表現
  - 完了 / 進行中 / 未着手 を視覚化
- **B-1 ビュー**
  - 16年 × 3問のグリッド（§4.3）
  - 年2問選択 = 計32問の対象を識別
  - 各セルに正答率 と「3設問中N正答」
- **共通**：消化工数 / 完答率 / 平均所要時間 / 次回予定 の KPI 帯

### 5.6 S6 週次レビュー

§6 のフローを3ステップに分解。

| ステップ | 入出力 |
|---|---|
| ① 測る | §6.1 の指標を直近4週分テーブル表示 / 所感メモ / エスカレーション基準（§6.5）の自動判定 |
| ② ふりかえる | §6.2 のミス3分類の件数入力 / 連続最多カテゴリ警告 |
| ③ うごかす | §6.3 の翌週打ち手（最大3つ）入力 / §6.4 のバッファ運用候補の提示 |

※ 計画書 §6.2 末尾にあった「科目B-2 4観点による自己採点」は本アプリでは採用しない（計画書からも削除済み）。

### 5.7 S7 月次レビュー

§7 のフロー全体を1画面で扱う。

- §7.1 週次指標の月次集計（傾向矢印付き）
- §7.2 区分（A-2/B-1/B-2）ごとの A/B/C 判定と「上位に上げる条件」
- §7.4 残工数の逆算と削減順
- §7.5 「やる / やらない」3つずつの宣言

### 5.8 S8 弱点マップ

- 状態カンバン：`解消 / 改善中 / 停滞 / 未着手`（§7.3）
- 各エントリは `{ 論点, 科目, ステータス, 直近の対策メモ }`
- **エスカレーション**：「未着手」が3件以上溜まると、翌月の重点候補として S7 に渡せる導線を出す

## 6. データモデル（論理）

```
Plan        ← 計画書由来（読み取り専用、§3, §4 をそのまま反映）
  ├ Subject (A-2, B-1, B-2)
  ├ B2Theme (23件; §4.1, §4.2)
  └ B1Year  (16年度 × 3問; §4.3)

Card        ← 科目A-2 (§5.1)
  c (通し番号), year, q, introDay

Problem     ← A-2 問題コンテンツ（外部データソース、初期は空で1レコードずつ追加）
  cardC, stem, choices[],                   // 選択肢はテキストまたは画像参照
  figures[ {position, src, caption?} ],     // 本文中や選択肢に振りうる図
  answerIndex, commentary, tags[]

Appearance  ← Card × 出現回数（§5.1 ②から計算可能、永続化は不要）
  cardC, day, rep (0=導入)

Task        ← 日次キューの1エントリ
  date, subject, ref(Card.c or B2Theme.id+kosuu or B1Year+q),
  rep, estimateMin, status(pending/done/skipped)

Attempt     ← Task の実施結果
  taskRef, grade, elapsedMin, mistakeCategory?, memoText?

WeakPoint   ← 弱点メモ
  id, topic, subject, status, note, linkedAttempts[]

WeeklyReview / MonthlyReview
  期間, 指標値, ミス分類件数, 打ち手[], 区分判定, …
```

- **派生量**：週次指標は `Attempt` から算出（永続化はキャッシュとして可）。
- **時間軸**：計画 Day（1..228）と西暦日付の双方を持つ。換算ルールは計画書 §3 を基準とする。

## 7. 機能要件（横断）

| FR | 内容 |
|---|---|
| FR-1 | 本日のタスクキューを自動生成できる：A-2は §5.1 の導入式と復習間隔（公比2）から、B-1/B-2 は実施曜日・進行状況から確定 |
| FR-2 | タスクの自己採点結果を記録し、次回出現を更新できる（§5.1 ② のルールに従う） |
| FR-3 | 誤答に対し3カテゴリ（§6.2）のいずれかのタグを付与できる |
| FR-4 | 気づきメモは1クリックで弱点メモに昇格できる |
| FR-5 | 週次指標（§6.1 の各値）を Attempt から自動集計し、目標値との差を視覚化できる |
| FR-6 | エスカレーション基準（§6.5）に該当した時点で警告を出せる |
| FR-7 | 月次レビュー時、§7.4 の残工数式（残週数 × 週工数）を自動表示し、超過時は §7.4 の削減順候補を出せる |
| FR-8 | 弱点マップで「未着手」が一定数を超えた時、翌月重点（§7.5）への組み込みを提案できる |
| FR-9 | **コーチメッセージはルールベースで生成する**。LLM や外部サービスに依存しない |
| FR-10 | **A-2 問題コンテンツは外部データソースから読み込む**。初期は空でよく、後から追加可能。追加手段は GUI でなくてもよい（ファイル追加 / CSV 等）。問題本文中の図、選択肢に含まれる図を含むこと |
| FR-11 | **スマホでも閲覧と最低限の記録ができる**：タスク完了、セルフ採点、ミス分類、気づきメモの追記 |
| FR-12 | **記録はクラウドで同期**され、PC とスマホのどちらからも同じ状態を見られる。同一レコードの競合は「最終更新を採用」、追記型のものはマージとする |

## 8. 非機能要件

| カテゴリ | 要件 |
|---|---|
| 利用環境 | **PC・スマホ両対応**。レスポンシブ設計または PWA 化を含めて検討。スマホではタッチしやすいコントロール（最低44px）、読みやすいテキストサイズ、スクロール負荷の低いレイアウトを採る |
| 永続化 | **クラウド同期を採用**し、デバイス間で記録を共有する。オフライン時はローカル（`localStorage` 等）に一時保存し、オンライン復帰時に同期する。具体のバックエンドは今後選定 |
| パフォーマンス | 画面遷移 < 300ms。S4 のグリッド描画は 4週28日 × 平均5枚 ≒ 140要素を1フレームで |
| バックアップ | エクスポート（JSON / Markdown）機能を将来実装 |
| 信頼性 | 操作ミスで履歴が消えないこと（記録は追記型を基本に） |

## 9. 設計原則

1. **計画書を編集させない**：計画書はソース・オブ・トゥルース。アプリの編集機能を増やして計画書と乖離させない。
2. **画面ごとに「考える単位」を揃える**：日次=S2、A-2の予定=S4、過去問の消化=S5、振り返り=S6/S7。混ぜない。
3. **入力は最小、集計は自動**：採点・所要時間など人が必ず打ち込むものだけを入力させ、指標は計算で出す。
4. **AI Slop を避ける**：装飾的なグラフ・スコア・スタッツを増やさない。意思決定に使えない数字は出さない。
